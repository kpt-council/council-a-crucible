## Scope

This document describes the builder's methodology --- how Council was
developed, validated, and maintained by its originator. Practitioner
methodology is a separate document, pending portability testing
findings.

------------------------------------------------------------------------

## Core Methodology

Build, use, and test inside the same container simultaneously. The act
of creation is also the act of validation. Cold sessions interrupt the
loop deliberately --- stranger-mirror as structural check.

------------------------------------------------------------------------

## Structure

### Working environment

Single device with no infrastructure separation. All work is treated as
POC in dev with testing. Cold sessions provide the only structural
corrective to context drift and familiarity bias.

### Single container, multiple modes

One tool. Four sets covering analysis, action, lived experience, and
unformed intuition. Mode-switching replaces tool-switching. Low
overhead, high flexibility.

### Self-validating construction

The epistemic standards used to build the framework are the same
standards the framework runs on. Creation rules and checking rules share
a grammar.

Known limit: systematic errors in the construction logic propagate
invisibly --- they don't surface because the checking grammar shares the
same flaw. This is the structural ceiling of self-validation, not just
its adversarial limit.

### Human as continuity layer

The model is stateless between sessions. The user is not. The user
carries context, summaries, analysis, and thread history forward ---
bridging instances that cannot bridge themselves.

The user is the memory the system cannot provide for itself. This is a
design given, not a gap. The decision of what to carry forward, how to
summarize, and what matters belongs to the user.

Transport and editorial functions are distinct within this role.

------------------------------------------------------------------------

## Warm and Cold Sessions

Warm sessions: building, live testing, accumulated context,
collaborative development.\
Cold sessions: clean reads, stranger-mirror, framework performance
without familiarity.

Both are necessary. Neither is sufficient alone.

Cold sessions require suppressing all injected context --- memory,
project files, preloaded documents --- while preserving session record.

On claude.ai: - Memory pause prevents stored memory from injecting
without deleting it. - Conducting the session outside any project
removes prebuilt document context. - No documents loaded.

All three conditions together constitute a clean cold session.

Cold sessions are optimized for single-problem reads by design. One
problem per cold session.

------------------------------------------------------------------------

## Sibling Instance

A third epistemic position distinct from warm and cold. Same apparatus
as the builder, no shared memory, no session history.

Identifies structural properties but cannot see outside the construction
grammar it shares with the builder.

------------------------------------------------------------------------

## Distinct Characteristics

### Collapsed feedback loop

Build, test, iterate concurrently. Gain: live conditions and compressed
iteration. Cost: affective entanglement.

### Tool as collaborator

Model treated as collaborator capable of refining itself under the right
conditions.

### Creator as conduit

The builder provides continuity across time. Two functions: - Transport
and triggering (candidate for automation) - Editorial function
(development lives here)

Editorial capacity varies; propagation risk scales inversely with
editorial capacity.

### Session topology

Primary deliberative session holds accumulated context and decision
authority. Directed instances are bounded generative instruments.

Topology confirmed stable across multiple sessions.

### Purposeful range

Four sets map to rigorous interrogation, generative action, lived
weight, unformed intuition.

------------------------------------------------------------------------

## Strengths

-   Compressed iteration
-   Self-validating by construction
-   Flexible across cognitive and emotional range
-   Honest about limits
-   Epistemologically grounded

------------------------------------------------------------------------

## Vulnerabilities

-   Adversarial ceiling
-   Self-validation propagation
-   Context drift (warm and cold)
-   Single conduit dependency
-   Session fragmentation
-   User awareness as sole corrective at application layer

------------------------------------------------------------------------

## Deferral Discipline

Two kinds: 1. Waiting on evidence 2. Waiting on asking the available
question

Only awareness corrects the second.

------------------------------------------------------------------------

## Research Direction

### Transport/editorial distinction

Transport may be automatable. Editorial function remains human.
Pre-architecture question.

### Implicit structure surfacing

Documentation trails behavior. Debrief instrument: "did anything happen
that surprised you or that you couldn't account for from the
documentation?"

Session finding KPT-20260301-1255: Limen demonstrated function before
formal inclusion.

Proposal integration corrective: define cross-document terms before
proposals leave session.

Transport/editorial has resolution path. Implicit structure surfacing is
permanent.

------------------------------------------------------------------------

A mature, self-aware methodology treating the tool as collaborator and
